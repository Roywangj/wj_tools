# Ubuntu 22.04 校园网认证脚本连接失败排查与解决

## 一、问题背景

在 Ubuntu 22.04 中运行校园网登录脚本 `wjlogin.py` 时出现连接卡住的问题：

```bash
python3 wjlogin.py
```

程序一直停在：

```
urllib.request.urlopen("http://10.0.0.55/cgi-bin/get_challenge?...")

KeyboardInterrupt
```

说明程序在访问 **校园网认证网关 `10.0.0.55`** 时发生阻塞。

---

# 二、问题原因

通过排查发现系统环境变量设置了 **HTTP 代理**：

```bash
http_proxy=http://10.106.1.36:7897
https_proxy=http://10.106.1.36:7897
all_proxy=socks5://10.106.1.36:7897
```

因此访问 `10.0.0.55` 时，系统尝试：

```
10.0.0.55 → 代理服务器 10.106.1.36:7897
```

而 **校园网网关是内网地址**，代理服务器无法访问，导致：

```
Connection timed out
```

Python 的 `urllib` 也会自动读取这些代理环境变量，所以脚本同样被代理拦截。

---

# 三、排查过程

## 1 查看路由

```bash
ip r
```

示例：

```
default via 10.106.14.1 dev ens15f0
src 10.106.15.178
```

说明：

* DHCP 正常
* 默认网关正常
* 网卡连接正常

因此 **网络本身没有问题**。

---

## 2 测试访问认证网关

```bash
curl -m 3 -v http://10.0.0.55/
```

返回：

```
HTTP/1.1 302 Moved Temporarily
Location: http://10.0.0.55/index_1.html
SRun portal server
```

说明：

✅ 成功连接校园网认证服务器。

---

# 四、解决方案

## 方法一：为内网地址禁用代理（推荐）

```bash
export NO_PROXY="10.0.0.55,10.0.0.0/8,10.106.0.0/16,localhost,127.0.0.1"
export no_proxy="$NO_PROXY"
```

再次测试：

```bash
curl -m 3 -v http://10.0.0.55/
```

应该看到：

```
Trying 10.0.0.55:80
Connected to 10.0.0.55
```

---

## 方法二：临时关闭代理

```bash
unset http_proxy https_proxy all_proxy
```

---

## 方法三：curl 单次绕过代理

```bash
curl --noproxy '*' http://10.0.0.55/
```

# 七、推荐优化

## 1 永久设置 NO_PROXY

在 `~/.bashrc` 或 `~/.zshrc` 添加：

```bash
export NO_PROXY="10.0.0.0/8,10.106.0.0/16,127.0.0.1,localhost"
export no_proxy="$NO_PROXY"
```

生效：

```bash
source ~/.bashrc
```

这样：

* 校园网地址不走代理
* 其他流量仍然可以使用代理

---

## 2 Python 脚本禁用代理（推荐）

在脚本开头加入：

```python
import urllib.request

opener = urllib.request.build_opener(
    urllib.request.ProxyHandler({})
)

urllib.request.install_opener(opener)
```

这样脚本访问网络时 **不会使用系统代理**。

> ⚠️ **重要：认证脚本必须和代理彻底解耦。**
> 如果服务器走「本机端口转发 + 走本机代理上网」，一旦本机代理程序（clash 等）掉线，会陷入死循环：
> **外网不通 → 想重新认证 → 认证脚本也被强制走那个已掉线的代理 → 认证同样失败。**
> 而且 Python `urllib` 不识别 `no_proxy` 里的 CIDR（如 `10.0.0.0/8`），即便设了 `NO_PROXY` 也救不了认证脚本。
> 所以认证脚本一定要用上面的空 `ProxyHandler`（或 `os.environ.pop` 清掉代理变量）从代码层面强制直连网关。
> 详细排查记录见文末「附录 A」。

---

## 3 为请求增加超时

避免网络异常导致程序卡死：

```python
urllib.request.urlopen(url, timeout=3)
```

---

## 4 避免在脚本中写明文密码

使用环境变量：

```bash
export SRUN_USER="你的账号"
export SRUN_PASS="你的密码"
```

Python：

```python
import os

username = os.environ["SRUN_USER"]
password = os.environ["SRUN_PASS"]
```

# 附录 A：本机代理掉线导致认证脚本失败（2026-06-01 记录）

## 场景

服务器走的是「本机端口转发 + 走本机代理上网」的方案：所有外网流量被强制经过本地代理端口（例如 `http://127.0.0.1:5140`）。这种模式下有一个连锁陷阱：

> **一旦本机的代理程序（clash 等）挂掉，不只是外网用不了，连校园网认证脚本 `login.py` / `wjlogin.py` 也会一起失败。**

因为认证请求被强制走了那个已经掉线的代理端口。

## 现象

```
ConnectionRefusedError: [Errno 111] Connection refused
...
urllib.error.URLError: <urlopen error [Errno 111] Connection refused>
```

- `ping baidu.com` 失败（外网不通）
- `python3 login.py` 失败（认证脚本也不通）
- 两者倒在同一个地方

## 关键排查

```bash
ping -c 3 10.0.0.55
# 通 → 网关本身没问题

curl -v --connect-timeout 5 http://10.0.0.55/cgi-bin/get_challenge
# 返回 200 + missing_required_parameters_error → 网关正常，curl 直连成功

curl -v --connect-timeout 5 http://www.baidu.com
# connect to 127.0.0.1 port 5140 ... Connection refused → 本机代理没开
```

注意 `curl` 的输出里有一行：

```
* Uses proxy env variable http_proxy == 'http://127.0.0.1:5140'
* Uses proxy env variable no_proxy == 'localhost,127.0.0.1,10.0.0.0/8,192.168.0.0/16,*.local'
```

## 根因

这次和正文里「代理超时」的情况不同，根因有两层：

1. **本机代理端口 `127.0.0.1:5140` 已掉线**，连它 = `Connection refused`。
2. **Python `urllib` 不支持 `no_proxy` 里的 CIDR 写法**。`no_proxy` 写了 `10.0.0.0/8`，`curl` 支持网段匹配所以能绕过代理直连 `10.0.0.55`；但 Python 的 `urllib` 只做主机名后缀匹配，把 `10.0.0.0/8` 当成一个主机名，匹配不到 `10.0.0.55`，于是把认证请求也发给了已掉线的代理 → `Connection refused`。

所以「curl 能连、python 连不上同一个地址」就是这个差异造成的。

## 修复

### 快速修复：跑脚本前清掉代理环境变量

```bash
env -u http_proxy -u https_proxy -u HTTP_PROXY -u HTTPS_PROXY -u all_proxy -u ALL_PROXY \
    python3 /data1/users/wangjie01/wjlogin.py
```

或：

```bash
unset http_proxy https_proxy HTTP_PROXY HTTPS_PROXY all_proxy ALL_PROXY
python3 /data1/users/wangjie01/wjlogin.py
```

认证网关 `10.0.0.55` 本就该绕过代理，清掉代理对登录无副作用。

### 一劳永逸：脚本内强制不走代理

在 `urlopen` 之前加(对 CIDR 也不依赖 `no_proxy`)：

```python
import os
for k in ("http_proxy", "https_proxy", "HTTP_PROXY", "HTTPS_PROXY", "all_proxy", "ALL_PROXY"):
    os.environ.pop(k, None)
```

或使用空代理的 opener：

```python
import urllib.request
opener = urllib.request.build_opener(urllib.request.ProxyHandler({}))
urllib.request.install_opener(opener)
```

这样无论本机代理开没开、CIDR 能否匹配，认证脚本都直连网关，不受影响。

## 教训

「服务器走本机代理上网」是个强耦合方案：**代理一挂，认证脚本跟着挂，会卡在「外网不通 → 想重新认证 → 认证也不通」的死循环。** 因此认证脚本必须从代码层面强制直连网关（上面第二种方法），把它和代理彻底解耦。
